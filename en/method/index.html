PROCESS · 10 STEPS

How I work.

I take every site through ten steps. Not a rigid policy — a sequence that removes most of the problems before they appear.

Below — what I do at each step and why. After the brief, sign-off before each next step: so the outcome isn't a surprise.

01 Brief 02 Studying the niche 03 Strategy 04 Site structure 05 Copy 06 Design system 07 Layouts 08 Development 09 Quality control 10 Handover
  1. 01

    Brief

    First conversation. I learn the brief, the audience, the budget, the deadlines, the constraints. The clearer this frame — the less rework later.

    How exactly
    • A short questionnaire or a 30–40-minute call.
    • Project goals — specific, not "more clients".
    • Audience — who they are, what matters to them, what they live by.
    • Constraints: budget, deadlines, existing identity, what can't be changed.
    What usually breaks if you skip this
    • "Make it nice" without saying for whom.
    • Goals like "more sales" with no number or timeline.
    • Starting work before the brief is agreed — rework guaranteed.
  2. 02

    Studying the niche

    I look at 5–7 competitor sites. What's working, what isn't, what's missing. Where no one has gone — that's where we can go.

    How exactly
    • For each site I note: palette, typography, hero structure, copy tone, how they push for action.
    • If 4 of 5 do the same thing — that's the niche's anti-pattern, no point running the same direction.
    • If no one does something obviously useful — that's the opening.
    What usually breaks if you skip this
    • "All sites in this niche look the same" — usually means you looked at one.
    • Copying what looks "nice" at competitors — everyone ends up looking the same.
    • Ignoring what already works in the niche — repeating someone else's mistakes.
  3. 03

    Strategy

    Two or three directions for the project — each with its own tone, structure and promise. We discuss, we pick one. From there we work only within its frame.

    How exactly
    • For each direction — a short description: which audience it suits, what it promises, what risks it carries.
    • I recommend one. Client agrees or pushes back — we discuss.
    • Once chosen — we lock it in writing. Every later edit gets checked against this document.
    What usually breaks if you skip this
    • "Let's blend a bit from each" — you end up with no character.
    • Strategy as a formality, without real agreement — design stage breaks.
    • A choice without a written record — two weeks later no one remembers what we decided.
  4. 04

    Site structure

    Sitemap: which pages are needed, what's on each, in what order the blocks go. At this stage we see — many pages or two, multipage or landing.

    How exactly
    • Every page — justified by why it's there. If a page exists "because everyone has it" — we drop it.
    • For a landing — block sequence with logic: first attention, then convince, then call to action.
    • For multipage — transitions and nav menu thought through.
    What usually breaks if you skip this
    • An "About" page with nothing to say.
    • A blog with no plans to write for it.
    • A product catalog on a landing meant to sell one service.
  5. 05

    Copy

    I write the copy before sitting down to mockups. Meaning first — form after. This changes page quality a lot.

    How exactly
    • Tone is chosen upfront — calm, friendly, formal, technical. Overdoing one side loses the other part of the audience.
    • Every block has a goal: what the reader should understand or do after reading.
    • Buttons and links — specific verbs, not "learn more".
    What usually breaks if you skip this
    • Marketing words with no meaning: "revolutionary solution", "innovative approach", "the highest quality".
    • Text "we'll rewrite later" — later means never.
    • Design on lorem ipsum — structure built around text that doesn't exist yet.
  6. 06

    Design system

    Palette, fonts, base components. One document that the whole site is later built from. Without it every page drifts a bit from the next.

    How exactly
    • Palette — 2-3 options with different moods. Dark and light. Warm and cool.
    • Fonts — for headings and for body. Sometimes I add monospace for numbers and code.
    • Components: buttons, fields, cards, dividers. Each — in several states (default, hover, error).
    What usually breaks if you skip this
    • Inter font and black by default — lazy, not a decision.
    • Colors picked without contrast check — text unreadable on light areas.
    • Components invented "on the fly" — pages drift away from each other.
  7. 07

    Layouts

    Final pages — desktop and mobile. Assembled from design-system components and the copy written in step five.

    How exactly
    • Desktop first, then mobile — same structure, different placement.
    • I check each block with long and short text — sometimes a heading wraps and that shifts the layout.
    • Show to client, collect feedback, iterate until agreement.
    What usually breaks if you skip this
    • Desktop-only mockups — on phone everything falls apart.
    • Static images without states — client expected one thing, saw another in action.
    • Demo on one device — client opens on phone and is surprised.
  8. 08

    Development

    Building the site from approved mockups. Clean code, image optimization, responsive across all screens.

    How exactly
    • Stack chosen for the task: static site, multipage, e-commerce — different answers.
    • Images in modern formats (WebP, AVIF), sized correctly for each screen breakpoint.
    • Code compact, no extra libraries, so the page loads fast.
    What usually breaks if you skip this
    • Libraries added "because popular" — they bloat without benefit.
    • Mismatch with mockup: "almost like the mockup" = not done.
    • Responsive as an afterthought — layout redone at the end.
  9. 09

    Quality control

    Checklist before handover: load speed, accessibility for people with disabilities, SEO basics, cross-browser, form testing.

    How exactly
    • I run Lighthouse on the live domain — not localhost. If the score is under 90 — I find out why.
    • Check in Chrome, Firefox, Safari. On phone and on desktop.
    • I submit a test form to make sure the email arrives.
    • Every link — clicked, confirmed it goes where it should.
    What usually breaks if you skip this
    • "Works in Chrome" instead of testing in Safari — and you hear about the bug from the client.
    • Form submission doesn't arrive — and the client doesn't know leads are vanishing.
    • Lighthouse score checked on localhost, on prod it's different.
  10. 10

    Handover

    Deploy to the live domain, SSL setup, instructions on how to edit the text. After that the site runs without me.

    How exactly
    • Domain and hosting — I set them up, I don't leave the client to figure it out.
    • Short manual: how to change the phone number, add a news item, update an image.
    • If something needs changing six months later — I'm reachable, no separate support contract.
    What usually breaks if you skip this
    • Site handed over "as is", no manual — client is afraid to touch anything.
    • SSL forgotten — browser shows a security warning.
    • Disappearing right after handover — a small bug sits there for weeks.
